Frontend Testing Documentation - Sprint 3

Overview

For Sprint 3, we implemented Playwright automated tests for the Staff Portal. The focus was on verifying that the key functionalities work correctly, and that the user interface remains consistent, responsive, and bug-free.

Test Coverage

The following areas were tested in Sprint 3:

  • Staff Dashboard: Verified page routing, loading, and rendering of the dashboard components.
  • Chatbox Tests: Ensured that sending, receiving, and displaying messages works correctly.
  • Announcements: Checked creation, display, and deletion of announcements on the manager dashboard.
  • Incident Reporting: Verified submission, validation, and correct routing of incident reports.
  • Venue Selection: Tested dropdown selections and image uploads for venue management.

Session & Cookies

Since our tests require an authenticated session, we saved cookies from a logged-in manager account into tests/cookies.json. Each test loads these cookies before navigating to the target page:


const cookies = JSON.parse(fs.readFileSync('tests/cookies.json', 'utf-8'));
await page.context().addCookies(cookies);

This ensures all tests run under a valid session and eliminates failures caused by expired logins.

Running the Tests

To run all frontend tests and generate an HTML report:

npx playwright test tests/ --reporter=html

After completion, open the report using:

npx playwright show-report

Reliability & Robustness

The test cases were designed to only fail when there is a genuine frontend or backend issue. For example:

  • Incident report tests fail if the route or backend validation is misconfigured.
  • Announcement creation/deletion tests fail if there is a backend processing error.
  • Chat tests fail only if messages cannot be posted or retrieved correctly.

This ensures that failures point to real problems rather than flaky test implementations.

Additional Notes

Most major platform functionalities were covered, including dashboard rendering, staff interactions, messaging, announcements, incidents, and venue selections. This gives us confidence in the platform's frontend stability before deployment.
